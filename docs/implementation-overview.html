从「向导里怎么选」到「各客户端 YAML」：中间用统一模型承接，规则来源与渲染分开，便于扩展。 用浏览器直接打开本文件即可查看矢量图。

人话版：小白读完就能懂「这工具帮你干啥」

一句话：你不用自己啃那种全是英文、缩进还不能错的配置文件；像做问卷一样点几步，最后给你一份能直接导入 OpenClash、Mihomo、Sparkle 的成品。

换成生活比喻：你告诉店家「我要吃什么菜」（哪些网站、哪些 App 要走代理）、「送走菜路线」（走香港还是日本节点）、「食材从哪来」（机场订阅链接），店家帮你打包成一张「配送单」——在这里就是一份 YAML。

你实际只要在脑子里记住四件事：

  1. 我用啥设备上的代理？路由器还是电脑、用哪家客户端，开头选一下。
  2. 我的节点从哪来？把机场给的订阅链接贴上，工具会把节点收进来。
  3. 地区怎么分队？比如「日本」「香港」各自一组；可以手写过滤条件，也可以用「关键词包含/不包含」让工具帮你写规则。
  4. 上网流量怎么走？先勾选别人整理好的「网站/App 清单」（也可以从全库里搜，比如游戏、流媒体），再指定：这条清单走「AI 组」「流媒体组」还是默认代理——也就是「谁先过站、最后走哪国节点」。

和「直接抄别人配置」有啥区别？别人的文件里，组的名字、订阅的名字往往和你对不上，一导入就报错或根本跑不起来；这个工具是按你的订阅、你的分组习惯生成，少了不少对表改名的麻烦。

用户可见 / 向导与导出 统一模型（扩展客户端主要旁路分支） 生成 YAML 中的业务链路

文字版摘要（可复制）

  • 实现目标：把「我想怎么分流」变成 OpenClash / Mihomo / Sparkle 能直接用的配置，降低 YAML 手写门槛。
  • 向导侧：选设备 → 填订阅 → 配置地区节点组（正则/自动测速）→ 选规则（预设 + MetaCubeX 全量检索）→ 把规则分配到服务策略组 → 预览并导出。
  • 核心层：所有选择汇入 BuilderProject（组、订阅、规则集、规则、设置），先校验再渲染，保证结构一致、可存 JSON。
  • 适配层：同一模型经不同 Renderer 输出各平台 YAML；新平台主要新增渲染与能力声明，不必推翻向导。
  • 规则来源：内置包与在线目录解耦；全量库侧重「工具内可选 rule-set」，客户端 geo 更新仍由用户自有客户端负责。
  • 交付形态：网页 + 可选 Tauri 桌面（Windows 发布安装包/便携版）；导出在浏览器为下载，在桌面为系统保存对话框。
